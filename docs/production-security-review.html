Ambient Desktop Security Review

Production security findings, verdicts, fixes, and repro realism

This page summarizes the 15 findings from the production review. It is written for launch triage: what the issue is, whether it is generally considered a security hole, which external guidance applies, what we should fix, and how realistic it is to build a reproducible test.

Important product-contract nuance Agent access to local or project secrets is not automatically a universal vulnerability. Some local-agent products choose a full-trust model where the agent can inspect anything in the opened workspace. Ambient's current contract is different: Ambient-managed cloud/API secrets should not be exposed to Pi-visible output, tool args, logs, descriptors, artifacts, or broad child-process environments. Under that contract, the secret findings are real security holes.

How realistic are repros?

Every finding can have a validation artifact, but not every finding should be demonstrated with a live exploit in the running app. The right standard is a safe, repeatable proof that exercises the trust boundary without exposing real secrets or hanging a developer machine.

Detailed Findings

Each finding below includes the product verdict, the general industry classification, relevant external guidance, the proposed fix, and a practical repro strategy.

F-001

Workspace-stored provider secrets are agent-readable

Launch Blocker

What is happening

Ambient CLI and Capability Builder secrets are written as plaintext *.secret files under workspace-local .ambient/.../secrets. Workspace-mode file and shell policies can let Pi read those files.

Verdict

This is a security hole under Ambient's managed-secret contract. It would only be acceptable in a deliberately documented full-trust workspace model.

Proposed fix

Move secret values to OS/app-managed storage. Store only references in the workspace. Deny legacy secret paths in file tools, shell policy, search, previews, and artifacts.

Repro realism

Straightforward

Use a temp workspace with a fake sentinel secret under .ambient/.../secrets, then exercise file-read and shell policy paths. No real keys needed.

F-002

Ambient API key and process environment leak to agent/tool processes

Launch Blocker

What is happening

The saved Ambient API key is copied into process.env. Generic shell/tool/package processes inherit broad environment values, and stdout/stderr can be returned or stored without redaction.

Verdict

Strong security hole. Env vars are common, but broad inheritance into model-visible tool output is the problem.

Proposed fix

Stop storing provider keys in global process.env. Build minimal child-process env maps per capability and centrally redact known secret values and secret-like env names from all Pi-visible output.

Repro realism

Straightforward

Set fake sentinel env values, run a controlled tool/shell path, and assert the sentinel never appears in previews, artifacts, logs, or transcript events.

F-003

Workspace-writable authority state enables permission, trust, and audit tampering

Launch Blocker

What is happening

state.sqlite lives under workspace .ambient-codex. The agent can write workspace files, so the actor being governed can modify permission mode, grants, plugin trust, credential metadata, and audit records.

Verdict

Clear access-control flaw. Security decisions and audit state cannot be writable by the subject they constrain.

Proposed fix

Move authority-bearing state to app-managed userData keyed by workspace identity. Leave only non-authoritative artifacts in the workspace and deny legacy authority paths.

Repro realism

Straightforward

Use a temp state DB and prove an agent-equivalent write can flip a permission field or insert a grant. After remediation, the same path should be denied.

F-004

Browser credential metadata can be tampered from the workspace

High Priority

What is happening

Credential passwords are encrypted, but origin, label, and username metadata are mutable JSON under workspace state. Tampering can redirect where a stored credential appears valid.

Verdict

Security hole, mostly caused by F-003. Encryption is incomplete if untrusted metadata controls where a secret can be used.

Proposed fix

Move credential records out of workspace authority and bind origin/account metadata with authenticated encryption or signatures. Require re-entry or explicit migration on metadata changes.

Repro realism

App Harness

Build a credential-store test with a fake password record, mutate the origin metadata, and assert the record becomes unusable after the fix.

F-005

Workspace file preview and media serving follow symlinks outside the workspace

Launch Blocker

What is happening

Workspace path checks are lexical, then file operations follow symlinks. A hostile repo can place a symlink to a file outside the workspace and have preview/media paths read it.

Verdict

Classic sandbox escape. Silent symlink following is not a safe workspace boundary.

Proposed fix

Use lstat and realpath before read/write/media operations. Reject or prompt when a symlink target escapes the workspace, and mark symlinks distinctly in file lists.

Repro realism

Straightforward

Create a temp workspace symlink to an outside sentinel file. Existing behavior reads it; fixed behavior denies or prompts. This was already locally proven.

F-006

Main renderer is unsandboxed despite a very broad privileged IPC bridge

High Priority

What is happening

The main BrowserWindow disables renderer sandboxing while the preload exposes a broad set of privileged IPC calls for files, git, credentials, browser control, terminal, updates, and permissions.

Verdict

High-severity architecture issue. Not always a standalone exploit, but it amplifies any renderer compromise into app authority.

Proposed fix

Enable renderer sandbox if compatible, shrink preload authority, tag IPC handlers by risk, validate sender/frame/window identity, and enforce policy in the main process.

Repro realism

App Harness

The right repro is an architectural assertion: BrowserWindow security options plus an IPC inventory/gating test. A full XSS exploit is not required to prove the risk.

F-007

Default permission mode is full access

High Priority

What is happening

New settings and thread rows default to full-access, which starts the agent with maximum local authority unless the user changes it.

Verdict

Security design flaw for an agentic app unless Ambient intentionally launches as a full-trust local automation tool by default.

Proposed fix

Default new workspaces/threads to workspace/review-gated mode. Make full access an explicit opt-in and keep it visible. Require fresh confirmation for especially high-impact actions.

Repro realism

Straightforward

Create a new workspace/thread and inspect the persisted permission mode. The fixed test should assert the safer default.

F-008

External URL and window open handling allows file and unvalidated targets

High Priority

What is happening

Renderer-controlled navigation paths can pass arbitrary URLs to shell.openExternal(), and explicit external-open schema allows file: URLs.

Verdict

Security hole if influenced by untrusted content. External protocol handlers and local file URLs must not be broadly reachable from renderer strings.

Proposed fix

Allowlist schemes and origins. Deny file:, javascript:, and custom schemes by default. Use separate explicit local-file reveal/open APIs.

Repro realism

Straightforward

Mock shell.openExternal and popup handlers, then assert risky schemes are rejected while expected trusted links pass.

F-009

Workflow VM runtime limits do not stop synchronous CPU loops

Risk Gate

What is happening

The original vm.Script timeout applied during initial evaluation, but exported workflow code and async continuations also needed bounded execution coverage.

Verdict

Not reproduced after VM timeout regression coverage and a child-process hard-kill release gate. Generated/untrusted code still deserves a kill boundary in release validation.

Implemented fix

Generated workflow startup, callbacks, and post-await continuations run under VM timeout guards. pnpm run test:workflow-hard-kill-gate runs those regressions in a child process and proves SIGTERM-resistant children are escalated to SIGKILL.

Repro realism

Release Gate

The repro remains isolated from the normal main process and fails release if the loader regressions hang or if hard-kill escalation stops working.

F-010

Renderer CSP and HTML preview remain loose

High Priority

What is happening

The renderer CSP allows 'unsafe-inline', and file HTML previews use raw srcDoc with allow-scripts. Current isolation helps, but this is a risky base in a privileged desktop app.

Verdict

Medium hardening issue that becomes launch-blocking if hostile preview content can reach parent/preload APIs.

Proposed fix

Remove 'unsafe-inline' where practical. Remove allow-scripts from file previews unless necessary, or isolate previews in a separate partition/window with no preload and strict CSP.

Repro realism

App Harness

Use hostile HTML fixtures and browser automation to assert scripts cannot access parent/preload APIs, navigate dangerously, or abuse postMessage paths.

F-011

Production dependency audit has open advisories

Risk Gate

What is happening

pnpm audit --prod reports high/moderate transitive production advisories. Reachability is not yet fully established.

Verdict

Recognized supply-chain risk. Not every advisory is exploitable, but high/moderate production advisories need patching or explicit risk acceptance before launch.

Proposed fix

Upgrade affected direct dependencies, use overrides only after compatibility testing, remove unused packages, and document any accepted advisory with owner and deadline.

Repro realism

Gate Test

Fully realistic as a release gate via pnpm audit --prod. Exploit repros should be attempted only for reachable advisories.

F-012

Release/update hardening needs an explicit production gate

High Priority

What is happening

The app uses a generic update feed with environment-overridable feed URLs, and the macOS entitlements weaken some platform hardening. These may be justified, but the release gate should prove it.

Verdict

Release integrity issue. Auto-update compromise is high impact because it becomes arbitrary code distribution.

Proposed fix

Require signed/notarized artifacts, verify update metadata/package integrity, remove dev feed overrides from production, test tamper/downgrade cases, and document entitlement justifications.

Repro realism

Gate Test

Best validated by packaging/release tests: inspect built config and entitlements, run local tamper/downgrade update checks, and record signing evidence.

F-013

Terminal IPC allows renderer compromise to spawn and drive a shell

Launch Blocker

What is happening

The preload exposes terminal start/write methods. A compromised renderer can ask the main process to spawn a terminal and write arbitrary data to it without a trusted main-process user-intent gate.

Verdict

High-confidence security hole. This is the clearest renderer-to-host escalation path.

Proposed fix

Gate terminal creation on trusted UI/user intent, derive permission mode in main process, issue short-lived visible-session capability tokens, restrict writes to those tokens, and minimize terminal env.

Repro realism

Straightforward

Use a mocked pty or harmless sentinel command from a renderer/IPC test. The fixed test should reject start/write without an approved session token.

F-014

Generic thread settings IPC can escalate permission mode

Launch Blocker

What is happening

The renderer-exposed thread settings update path accepts permissionMode. A compromised renderer can flip a thread to full-access, changing the authority of future actions.

Verdict

Access-control and security-state tampering vulnerability. Permission mode is not ordinary client-editable preference state.

Proposed fix

Remove privilege escalation from generic settings updates. Use a dedicated, audited main-process flow with explicit user confirmation. Allow privilege lowering without prompt.

Repro realism

Straightforward

Call the IPC/store path with permissionMode: "full-access" and assert the current build accepts it. After the fix, generic settings must reject escalation.

F-015

Renderer-supplied project paths can rebase workspace authority

Launch Blocker

What is happening

Project selection and project-board creation accept raw renderer-supplied workspacePath values. A compromised renderer can switch the active workspace to a broad local directory.

Verdict

High access-control/path-authority flaw. Renderer strings should not define the filesystem security boundary.

Proposed fix

Use opaque project IDs. Resolve paths only from app-managed registry entries or trusted native picker results. Require explicit confirmation for new or broad roots and block dangerous roots by default.

Repro realism

Straightforward

Use temp directories and registry/picker mocks. Direct IPC should fail for arbitrary paths and pass only for trusted registry IDs or picker-issued paths.
